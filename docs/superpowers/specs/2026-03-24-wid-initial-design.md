# wid 初期設計

## 概要

`wid` は "what I'm doing" を記録するための新しい CLI プロジェクトです。

最初の目標は完全自動化ではありません。人間にも AI にも使いやすい、安定した記録の中核を先に作ることです。初期段階では人間は自由に記録し、AI エージェントは「どのタイミングで記録するか」という運用ルールに従って記録します。

`wid` の中心にあるのは「終わったことの報告」よりも、「今から何をやるか」「何に着手したか」を忘れないための記録です。複数プロジェクトを行き来していると、直前に何へ切り替えたかを見失いやすいため、その切り替え点を軽く残せることを最優先にします。

当初は `doing` を踏み台にする案もあったものの、初期実装の時点から `wid` は独立した実装にします。保存形式、パーサ、CLI の挙動は `wid` 自身で持ちます。

## 目的

- 複数プロジェクトをまたいで使える 1 つのグローバルログを持つ
- 人間や AI エージェントが、作業の切り替え時点で自然に記録できる程度に入力コストを下げる
- 専用ビューアがなくても、そのまま人間が読めるログにする
- CLI の表面積は最小から始める
- 将来の自動化、メタデータ追加、よりよい TUI/CUI に拡張できる余地を残す

## 初期スコープに含めないもの

- `doing` への依存
- Git hook 連携
- プロジェクト名の自動推定
- actor、project、tag などの必須構造化メタデータ
- フルスクリーンの対話 TUI
- `init` コマンドや設定コマンド

## プロダクトの最小形

初期版の `wid` は Rust 製の小さな CLI とし、まずは次の 9 つの振る舞いを持ちます。

- `wid` で全件表示する
- `wid add ...` で新しい未着手項目を追加する
- `wid now ...` で新しい記録を追加する
- `wid done` で最後の未完了項目を完了にする
- `wid done -i` で未完了項目を選んで完了にする
- `wid focus -i` で既存の未完了項目に集中を戻す
- `wid edit` で現在の項目または最新項目の本文を編集する
- `wid edit -i` で項目を選んで本文を編集する
- `wid note` で項目にノートを追記する
- `wid rm -i` で全項目から 1 件を選んで削除する

これはタスク管理ツールではなく、あくまで「いま何に着手したか」「次に何をやるか」を見失わないための活動ログです。

## 保存モデル

保存の正本は 1 つのグローバル Markdown ファイルです。

初期の書式は、できるだけ小さく保ちます。

```md
# wid log

## 2026-03-24

- [ ] 11:32 CI が落ちていたので修正
- [>] 12:10 実装方針を見直した

## 2026-03-25

- [ ] 09:20 レビュー指摘を反映
```

### 書式上の前提

- ファイル形式は Markdown
- エントリは日付見出し `## YYYY-MM-DD` ごとにまとまる
- 各エントリは `- [ ] HH:MM summary`、`- [>] HH:MM summary`、`- [x] HH:MM summary` のいずれか 1 行の箇条書き
- `[ ]` は未完了だが今は触っていない状態、`[>]` は今取り組み中、`[x]` は完了を表す
- `[>]` はログ全体で最大 1 件だけ許可する
- summary は自然文で、人間がそのまま読める

### この形式を選ぶ理由

- どのエディタでもそのまま読める
- 追記処理が単純で壊れにくい
- 将来のパースが容易
- 早い段階で構造を入れすぎずに済む

## CLI モデル

### `wid`

デフォルトコマンドです。グローバルログの全件を表示します。

初期段階ではログ件数がまだ少ない想定なので、まずは全件表示を標準動作にします。件数が増えて全件表示がノイジーになった時点で、件数制限やフィルタを追加します。

例:

```bash
wid
```

### `wid now [TEXT...]`

現在のローカル時刻を使って、新しい 1 件を追記します。

引数が与えられた場合は、残りの引数を空白で連結して summary として保存します。

例:

```bash
wid now CI が落ちていたので修正
wid now レビュー指摘を反映
```

### `wid now`

引数がない場合は対話入力モードに入り、1 行の summary を入力します。

これは `(` や `&` のようなシェル上で扱いづらい文字を含む場合や、単に引用符なしで気軽に書きたい場合の逃げ道です。

例:

```bash
wid now
```

振る舞い:

- 1 行の入力を促す
- `Ctrl-C` で安全に中断できる
- 空入力はエラーとして拒否する

`wid now` は「これから取りかかること」や「今切り替えた作業」を記録するコマンドとして扱います。完了報告を書くこと自体は禁止しませんが、初期設計では主目的にしません。

追加時の保存形式:

```md
- [>] 11:32 CI が落ちていたので修正
```

振る舞い:

- `wid now` は新しい項目を常に `[>]` で追加する
- 追加前に既存の `[>]` があれば、自動で `[ ]` に戻す
- 人間にとって「今からやること」を 1 件だけ明示する入口とする

### `wid add [TEXT...]`

現在のローカル時刻を使って、新しい 1 件を未着手項目として追記します。

引数が与えられた場合は、残りの引数を空白で連結して summary として保存します。

例:

```bash
wid add 次にレビューする候補をメモ
```

追加時の保存形式:

```md
- [ ] 11:35 次にレビューする候補をメモ
```

振る舞い:

- `wid add` は新しい項目を常に `[ ]` で追加する
- 既存の `[>]` は変更しない
- 引数なし時は `wid now` と同じく 1 行の対話入力 fallback を持つ
- backlog 的に積みたい項目の入口とする

### `wid done`

最後の未完了項目を完了にします。

初期版では引数もオプションも持ちません。`[>]` が存在すればそれを優先して `[x]` にし、`[>]` がなければログ全体を新しい順に見て、`[ ]` になっている最後の項目を 1 件だけ `[x]` に更新します。

例:

```bash
wid done
```

更新前:

```md
- [>] 11:32 CI が落ちていたので修正
```

更新後:

```md
- [x] 11:32 CI が落ちていたので修正
```

振る舞い:

- すでに `[x]` になっている項目は飛ばす
- `[>]` があればその 1 件を優先して更新する
- `[>]` がなければ最後の `[ ]` だけを更新する
- 未完了項目がなければエラーにする

完了時刻は保存しません。`wid` では「いつ着手したか」を思い出せることを優先し、完了は状態としてだけ扱います。

`wid done` は `wid now` の対になる最小完了コマンドです。`doing finish` のような検索や複数件処理は、必要になるまで入れません。

### `wid done -i`

未完了項目をインタラクティブに選んで完了にします。

`wid done` は最後の未完了項目だけを閉じる最小コマンドですが、複数の作業を行き来していると「最後の 1 件」以外を閉じたい場面があります。そのため、`wid done -i` では全ログから未完了項目を集め、`wid` の表示順と同じく古い順に選択できるようにします。

例:

```bash
wid done -i
```

一覧の対象:

- 全ログにある未完了項目
- `[ ]` になっている項目だけを含む
- 表示順は `wid` の表示順と同じにする

一覧の表示内容:

- 1 行につき 1 項目
- `YYYY-MM-DD HH:MM summary` の形で表示する
- 初期選択位置は先頭、つまり一覧の最上段の項目にする
- 描画は `ratatui` ベースの単一カラム TUI とする
- 画面全体を切り替えるのではなく、現在のシェル画面を残したまま下部数行にインライン表示する
- 選択中の項目は背景色と左側アクセントで視認しやすくする
- 下部にキーガイドを出し、Omarchy の `fzf` 風に落ち着いた配色で表示する

操作:

- `j` または下矢印で次の項目へ移動する
- `k` または上矢印で前の項目へ移動する
- `Enter` で選択中の項目を完了にする
- `q` または `Esc` でキャンセルする

確定時の更新:

- 選択した 1 件だけの `[ ]` を `[x]` に変える
- 既存の保存形式は変えない
- 更新後の表現は通常の `wid done` と同じにする

未完了項目が存在しない場合は一覧を開かず、通常の `wid done` と同じくエラーにします。

### `wid focus -i`

既存の未完了項目から 1 件を選び、今取り組み中の項目として `[>]` にします。

`wid add` は未着手項目を積むコマンドで、`wid now` は新しい作業をそのまま開始するコマンドです。すでにある項目へ戻って作業を再開したいときには別の入口が必要です。そのため、`wid focus -i` を「既存項目へ集中を戻す」ための最小コマンドとして用意します。

例:

```bash
wid focus -i
```

一覧の対象:

- 全ログにある `[ ]` 項目
- 現在の `[>]` 項目があれば、それも一覧に含める
- `[x]` の項目は含めない
- 表示順は `wid` の表示順と同じにする
- 初期選択位置は、`[>]` 項目があればそこ、なければ先頭にする

確定時の更新:

- 選択した 1 件だけを `[>]` に変える
- 既存の `[>]` があれば、その項目は先に `[ ]` に戻す
- 更新後も `[>]` はログ全体で最大 1 件に保つ
- 既存の `[>]` をそのまま選び直した場合は no-op とする

項目が存在しない場合は一覧を開かずエラーにします。

### `wid note`

既存の項目に、自由文のノートをぶら下げて追記します。

`wid` の主眼は「今どこに意識を向けているか」を保つことですが、実際の作業中には短い判断メモや補足を残したくなることがあります。そのため、各項目の直下に自由文ノートを複数付けられる最小コマンドとして `wid note` を用意します。

例:

```bash
wid note picker では状態が見えないので削除前に不安がある
wid note
```

保存形式:

```md
- [>] 08:01 rm -i でアイテムの [ ] と [x], [>] も表示するようにする。
  - picker では状態が見えないので削除前に不安がある
  - rm -i だけ状態付き表示に寄せる
```

ルール:

- ノートは親項目直下の `  - free text` 形式にする
- インデントは 2 スペースに固定する
- ノート自体には checkbox を持たせない
- 1 項目に複数ノートを持てる
- ノートの並び順は追記順にする

対象の決め方:

- 現在の `[>]` 項目があれば、その項目に付ける
- `[>]` がなければ、最後の `[ ]` 項目に付ける
- どちらもなければエラーにする

入力経路:

- `wid note TEXT...` は引数を空白連結して 1 件追加する
- `wid note` は引数なしなら 1 行対話入力に入る

### `wid edit`

既存の項目の summary を 1 行だけ編集します。

`wid` は「今どこに意識を向けているか」を保つログですが、実際には typo の修正や、あとからよりよい表現へ言い換えたくなることがあります。そのため、まずは親項目の summary だけを対象にした最小編集コマンドとして `wid edit` を用意します。ノートの編集や state の変更は初期版では扱いません。

例:

```bash
wid edit
```

対象の決め方:

- 現在の `[>]` 項目があれば、その項目を編集する
- `[>]` がなければ、ログ全体で最後の項目を編集する
- 項目が存在しなければエラーにする

編集内容:

- 変更するのは summary だけに限定する
- `time`、checkbox state、ノートは変更しない
- 空文字への更新は拒否する

入力経路:

- 対象の現在値を初期値として 1 行編集に入る
- 確定した文字列でその項目の summary を置き換える
- `Ctrl-C` では安全に中断できる

### `wid edit -i`

全項目から 1 件を選び、その summary を 1 行編集します。

`wid edit` のデフォルト対象は「いま触っている項目」または「最後の項目」ですが、古い項目をあとから言い換えたい場面もあります。そのため、`wid edit -i` では一覧から編集対象を選んで、その場で 1 行だけ編集できるようにします。

例:

```bash
wid edit -i
```

一覧の対象:

- 全ログにある全項目
- `[ ]`、`[>]`、`[x]` のすべてを含む
- 表示順は `wid` の表示順と同じにする
- 描画は `wid done -i` と同じ `ratatui` ベースの picker を共有する
- picker はシェル画面の文脈を残したままインライン表示する

一覧での操作:

- `j` または下矢印で次の項目へ移動する
- `k` または上矢印で前の項目へ移動する
- `Enter` で選択中の項目の編集に進む
- `q` または `Esc` でキャンセルする

編集の操作:

- 選択した項目の現在の summary を初期値として 1 行編集に入る
- 空文字への更新は拒否する
- 確定した文字列で summary だけを書き換える
- `time`、checkbox state、ノートは変更しない

項目が存在しない場合は一覧を開かずエラーにします。

### `wid rm -i`

全ログの全項目から 1 件を選んで削除します。

`wid now` では気軽に記録できることを重視するため、誤入力や書き間違いが起きることがあります。そのため、`wid rm -i` は「後から 1 件だけ取り除く」ための最小削除コマンドとして用意します。

例:

```bash
wid rm -i
```

一覧の対象:

- 全ログにある全項目
- 未完了項目と完了済み項目の両方を含む
- 表示順は `wid` の表示順と同じにする

一覧の表示内容:

- 1 行につき 1 項目
- `YYYY-MM-DD HH:MM summary` の形で表示する
- `[ ]` と `[x]` のどちらの項目も表示対象に含める
- 初期選択位置は先頭、つまり一覧の最上段の項目にする
- 描画は `wid done -i` と同じ `ratatui` ベースの picker を共有する
- picker は alternate screen を使わず、シェル画面の文脈を残したまま下部に表示する

一覧での操作:

- `j` または下矢印で次の項目へ移動する
- `k` または上矢印で前の項目へ移動する
- `Enter` で選択中の項目の削除確認へ進む
- `q` または `Esc` でキャンセルする

確認の操作:

- 選択した 1 件を表示して `Delete this entry? [y/N]` 相当の確認を出す
- `y` を押したときだけ削除する
- `n`、`q`、`Esc`、その他のキーでは削除しない
- 削除しない場合は一覧へ戻るか、そのまま安全に終了する

削除時の更新:

- 選択した 1 行だけをファイルから削除する
- 既存の保存形式は変えない
- まずは空になった日付見出しの自動削除までは行わない

項目が存在しない場合は一覧を開かずエラーにします。

## 人間と AI の使い分け

初期版では、人間用と AI 用の違いはファイル形式ではなく運用ルールで分けます。

人間も AI も同じファイルに、同じ見た目の形式で書き込みます。違いは「どういう節目で書くか」だけです。

- 人間は自由に好きなタイミングで書いてよい
- AI エージェントは作業の切り替えや着手の節目で書く

AI にとって適切な節目の例:

- これから CI 修正に入る
- これからレビュー指摘の反映に入る
- ここからリファクタに着手する
- 次の実装塊に切り替える

これにより、ファイル形式を最小に保ったまま、AI 側だけ運用で規律を持たせられます。あとで振り返るためのログでもありますが、主眼は「今どこに意識を向けているか」を失わないことです。完了時には必要に応じて `wid done` で対応する 1 行を閉じます。

## AI スキルの位置づけ

AI 向けの判断ロジックは CLI 本体ではなく、別のスキルとして CLI の上に載せます。

スキルの責務:

- どの作業開始・切り替えの瞬間が記録に値するかを判断する
- 粒度を細かくしすぎず、意味のある着手単位に保つ
- 初回の数回だけ、人間に粒度確認を入れる
- 内容が決まったら `wid now ...` を呼ぶ

CLI の責務:

- 入力を受け取る
- Markdown 形式に整形する
- 安全に保存する
- 保存済みの履歴を表示する

この分離により、将来エージェントの挙動が変わっても CLI 契約は安定させやすくなります。

## ログの既定保存場所

ログは現在のプロジェクト配下ではなく、安定したグローバルパスに置きます。

初期の既定値としては、次を推奨します。

```text
~/.local/share/wid/log.md
```

理由:

- プロジェクト横断で使える
- `wid` の用途に合っている
- このリポジトリ自体を汚さない
- 将来、設定や環境変数で上書きしやすい

上書き機構そのものは初期スコープではまだ実装しなくてよいですが、コード構造としては後から入れやすくしておきます。

## パースと書き込みの方針

初期版では、賢すぎる再整形よりも、安全で予測可能な振る舞いを優先します。

推奨方針:

- ファイルがあれば既存内容を読む
- その日の `## YYYY-MM-DD` 見出しがあれば、その下に追記する
- なければ末尾に新しい日付セクションを足す
- 既存テキストはできるだけそのまま保つ

これにより、不要なフォーマット差分を避けつつ、手編集との共存もしやすくなります。

## エラーハンドリング

初期版の挙動はシンプルに保ちます。

- ログファイルがなければ初回書き込み時に作成する
- 対話入力が空ならユーザー向けエラーにする
- 古い内容に想定外の書式が混ざっていても、破壊的な再整形はしない
- `Ctrl-C` で対話入力を中断した場合は何も書かない

ファイル内に想定外の構造が含まれる場合、パースが必要な表示系処理は保守的に失敗してよい一方で、`wid now` は可能なら安全に追記できるようにします。

## Rust を選ぶ理由

Rust はこのプロジェクトと相性がよいです。

- CLI の体験を丁寧に作りやすい
- 単体バイナリとして配布しやすい
- 長期保守に向いている
- パースや将来のリッチな端末表現を安全に育てやすい

初期に想定するライブラリ:

- `clap`: CLI 引数定義
- `anyhow` または `miette`: ユーザー向けエラー処理
- 対話入力はまず標準入出力で十分
- `ratatui` + `crossterm`: 最小のインタラクティブ picker

## MVP の完了条件

次の条件を満たしたら、初期実装は完了とみなします。

- `wid` を実行するとグローバルログの全件が表示され、いま何に取り組んでいるかを思い出せる
- `wid add some text` を実行すると、その日の見出しの下に backlog 用の項目が `- [ ] HH:MM some text` として追記される
- `wid add` を実行すると簡単な対話入力で 1 件追加できる
- `wid now some text` を実行すると、その日の見出しの下に「これからやること」や「今切り替えた作業」が `- [>] HH:MM some text` として追記される
- `wid now` を実行すると簡単な対話入力で 1 件追加できる
- `wid done` を実行すると `[>]` を優先して `[x]` にできる
- `wid done -i` を実行すると未完了項目を選択して `[x]` にできる
- `wid focus -i` を実行すると既存の `[ ]` 項目を 1 件選んで `[>]` にできる
- `wid rm -i` を実行すると全項目から 1 件を選んで確認後に削除できる
- ログファイルがなければ自動作成される
- 保存されるファイルが、人間に読める Markdown として保たれる
